GOAL
  A Settings "Models" builder where a user declares their state variables, then per variable types:
    (1) the DETERMINISTIC RHS   -- the right-hand side of dx/dt = ... (first-order, may be nonlinear),
    (2) the WHITE-NOISE STRENGTH -- the parameter combination for the delta-correlated noise (white noise
        only for now); the app translates it to the solver's noise scale,
    (3) the EXTERNAL FORCING     -- one per variable, chosen from {spring, sinusoidal, step, triangular,
        exponential}, with the user typing the parameter names that force needs.
  The app parses this into a runnable model, SAVES it to Resources/Models/, and lists it in EVERY model
  dropdown (Simulate, Parameter Inference Config, FDT) alongside the 3 hardcoded defaults (BP, NADROWSKI,
  HOPF), which stay hardcoded. One equation-row per variable (no comma-separated systems) keeps input easy.

FEASIBILITY  —  FEASIBLE. Difficulty rises by app section:
    SIMULATE   EASY   -- the simulate path is already model-agnostic (see CONTRACT). v1 lives here.
    SBI        MEDIUM -- the numerics are model-agnostic, but a user model must meet a rescale/forcing/
                        Prior CONTRACT (below). Big scientific payoff; separate phase.
    FDT        HARD   -- the effective-temperature formula hardcodes Nadrowski params n, beta
                        (core/FDT/fdt_pipeline.py:126-127). Needs a model-specific T_eff normalization.
    REDUCTION  EXCLUDE-- core/Reduction is an intrinsic Nadrowski->Hopf normal-form map (reduce.py hardcodes
                        NWK_KEYS). Do NOT offer Reduction for user models.
  The enabling facts: models are DUCK-TYPED numeric torch classes (no base class to satisfy), and sympy is
  already pinned (requirements.txt:20) but COMPLETELY UNUSED -- a clean slate for the parser/codegen.

THE MODEL/SOLVER CONTRACT  (what a generated model must satisfy; verified against the 3 live models)
--------------------------------------------------------------------------------
  There is NO abstract base Model class -- core/Models/__init__.py is empty; models are plain classes whose
  "interface" is defined by how the solver + simulator call them. A generated UserModel must expose:
    * f(self, x, t) -> Tensor   : the DRIFT. x is (batch, d) torch; t is the INTEGER local step index (NOT
        physical time). Return (batch, d). FORCING is folded in here as an additive term read from
        self.force[:, ch, t] (there is no separate force method).
        Example: core/Models/hopf_model.py:40-50 (both channels driven), nadrowski_model.py:58-71 (only
        channel 0 driven), bp_model.py:44-52 (channel 0, scaled by 1/tau_hb).
    * g(self)  OR  g(self, x)   : the DIFFUSION -- a (batch, d) DIAGONAL amplitude vector (elementwise, no
        off-diagonal). Zero-pad noiseless channels (bp_model.py:54-59). Use g(self, x) + run with
        state_dep_drift=True for state-dependent (multiplicative) noise (nadrowski_model.py:78-91).
        !! Do NOT copy bp_model_steady.g() or the archived HarmonicOscillator -- they return full (batch,d,d)
           MATRICES, which the eager solver does NOT consume. The live convention is the (batch,d) vector.
    * self.force : a settable public (batch, n_channels, T) tensor. The Simulator overwrites it per time
        segment via a BARE attribute set (simulator.py:58) -- NOT the .force property (which rebuilds the
        model). Restored at simulator.py:66.
    * self.device : torch device; the solver does x0.to(sde.device). Must be device-agnostic (accept
        device/dtype, .to(...) all param tensors). Devices from config.detect_device()/cpu_device().
  SOLVER (core/Solvers/sdeint.py:33-67, the eager `euler`): Ito Euler-Maruyama
        x_{i+1} = x_i + f(x_i, i)*dt + g*dW*sqrt(dt),  dW ~ N(0,1).
    dt is DERIVED from (ts, n), never passed. state_dep_drift toggles g() once vs g(x) each step. There is
    also euler_compiled (CUDA torch.compile fast path, needs compiled_step/f_pure/compiled_params) -- OPTIONAL;
    __sols falls back to eager euler automatically (simulator.py:132-135). The per-step tqdm bar desc is
    config.SOLVER_BAR_DESC="step" (the GUI "Solver Performance" meter keys on it).
  CONSTRUCTION: simulators build the model POSITIONALLY:
        Model(*torch.unbind(params, dim=1), force, batch_size=..., device=..., dtype=...)
        (core/Simulator/nadrowski_simulator.py:16, hopf_simulator.py:16, bp_simulator.py:18)
    => the PARAM ORDER in the generated bounds file MUST equal the model constructor's positional arg order.
    _set_up_model is @abstractmethod (simulator.py:150) and calls bare exit() on bad construction -> SystemExit;
    the GUI catches it (simulate_runner.py:130-140). A UserSimulator(Simulator) must implement _set_up_model.
  OBSERVABLE: the app treats STATE COLUMN 0 as the observable (simulate_runner.py:194 emits res[1:,0,0];
    "state index 0 in every model"). A user model's primary observable must be variable index 0.
  What a model does NOT provide: dt (solver-owned), state dimension d (inferred from inits.shape[-1] /
    x0.shape), state names / initial conditions (supplied via the cell file's inits_dict).
  Everything is TORCH (no numpy in the hot path). numpy only appears in force construction.

PARSER + CODEGEN  (sympy; the core engineering)
--------------------------------------------------------------------------------
  * Parse each typed expression with sympy.parsing.sympy_parser.parse_expr in a LOCKED-DOWN namespace
    (explicit local_dict of allowed functions/symbols; no arbitrary Python eval). Collect free_symbols.
  * The user's DECLARED VARIABLE NAMES are the STATE variables; every other free symbol across all
    expressions is a PARAMETER. This inverts today's flow (params are discovered from FILES, not equations).
  * Codegen a torch-vectorized, device-aware UserModel: map each variable name -> state column index j
    (x[:, j]); emit f_i as a torch expression; add self.force[:, ch, t] to forced channels; return diagonal
    (batch, d) stacks for f and g. sympy.lambdify(..., modules=torch-compatible) or a small AST->torch
    emitter. Keep an ORDERED param list so torch.unbind(params, dim=1) lines up with the constructor.
  * NOISE TRANSLATION: the solver's g is literally the coefficient of dW. The white-noise strength D that
    the user types (spectral, <xi(t)xi(t')> = 2 D delta) maps to  g = sqrt(2 * D)  -- exactly what Nadrowski
    does (nadrowski_model.py:78-91: sqrt(2/(n*beta)), sqrt(2*tau_c*p*(1-p)/n)/tau). (Hopf instead uses g = sigma
    directly, BP uses g = eta/tau -- the convention is per-model today, so PIN it to sqrt(2D) for user models
    and say so in the UI.) Decide state-dependent noise support: additive-only in v1 is simplest; g(x) needs
    state_dep_drift=True (today hard-keyed to the string "nadrowski" at cli.py:45 -- give user models another
    way to set the flag, e.g. from whether the noise expr references any state variable).

FORCING  (only sinusoidal exists today; 4 of 5 are NEW; "spring" is special)
--------------------------------------------------------------------------------
  The ONE existing builder is core/SBI/pipeline.py:32-101 build_nondim_sin_force_tensor -> a precomputed
    (batch, n_channels, T) ND tensor (NOT a callable), sampled on the ND time grid, indexed by local step.
    F_dim(t)=amp*sin(2*pi*freq*t_dim+phase)+offset, then F_nd=(F_dim - f_offset)/f_scale. n_channels=2 iff
    "amp_y" present. Applied per-variable as an additive drift term (hopf_model.py:48-49 etc.). The GUI
    streaming mirror is simulate_runner.py:183-185 (sim.sde.force = force_chunk).
  ADD a shape dispatch that emits the SAME (batch, n_channels, T) tensor for:
    sinusoidal  = existing builder (params amp, freq, phase, offset).
    step        = offset + amp*(t >= t0)                    (params amp, t0, offset).
    triangular  = amp * triangle_wave(freq, phase) + offset (params amp, freq, phase, offset).
    exponential = amp * exp(+/- t/tau) + offset             (params amp, tau, offset).
  !! SPRING is NOT a time-drive: a Hooke force -k*(x - x0) is STATE-DEPENDENT and belongs in the drift f,
     not the precomputed force tensor. Either special-case it (inject -k*(x_i - x0_i) into that variable's
     RHS) or drop it from the "forcing" menu and tell users to put restoring terms in the deterministic RHS.
     Flag this clearly in the UI -- it behaves differently from the other four.
  SBI note: the forcing PRIOR builder orchestrator._build_forcing_prior (orchestrator.py:308-323) + the
    forcing param name set (amp/freq/phase/offset[/amp_y]) are sinusoid-shaped; SinPrior exists but is
    DEAD/unused. Generalizing forcing for SBI (v2) means generalizing these.

PARAMS / NONDIMENSIONALIZATION / PERSISTENCE
--------------------------------------------------------------------------------
  * The compatibility target is SimConfig (core/config.py:176-442), built by cli.make_sim_config
    (cli.py:362-389) from a Resources/{Bounds,Cells,Units}/<model>/ TRIPLE:
      Bounds/<model>/<cell>.txt  -- the param SET + ORDER (+ (lo,hi)); the single source of truth.
      Cells/<model>/<cell>.txt   -- VALUES only (a ground-truth point) + initial conditions.
      Units/<model>/units.txt    -- unit tokens (e.g. "nm ms pN Hz").
    Parsed by file_manager (file_manager.py:19-209) + cli._parse_cell (cli.py:277-334); params_dict etc. are
    OrderedDicts {name:(value,(lo,hi))}. So a user-model builder should EMIT these three files (the decoupled
    path) so make_sim_config + the whole pipeline flow UNCHANGED. Param order in Bounds must match the
    constructor (see CONTRACT). state_dep_drift is a string test today (cli.py:45) -- generalize.
  * EVERYTHING IS ND. Drift/noise are nondimensional; dimensional physics enters ONLY via the rescale params
    x_scale/t_scale/f_scale (config "# Dimensional Parameters" section), which are INFERRED, not derived.
    Nothing auto-nondimensionalizes. => v1 asks the user for ND equations + a t_scale/x_scale so the Simulate
    time axis (seconds) and displacement units label correctly (simulate_runner.py:201 redimensionalizes).
  * PERSISTENCE: save a model to Resources/Models/<name>.txt|json = {variables, per-var drift/noise/forcing
    exprs + chosen force types, param names+values, inits, x_scale/t_scale}. A startup loader reads them and
    registers each (see REGISTRIES). Also generate the Bounds/Cells/Units triple (or synthesize the
    OrderedDicts in memory). The 3 hardcoded models remain code-defined defaults.

GAPS / OPEN DECISIONS  (resolve at implementation -- these are the "what am I missing")
--------------------------------------------------------------------------------
  1. VALUES + INITIAL CONDITIONS, not just names -- to simulate you need a number for every parameter and an
     init per variable. The builder must collect them (and bounds too, if/when SBI is wanted).
  2. ND vs DIMENSIONAL -- require ND equations in v1 (nothing auto-nondimensionalizes); take x_scale/t_scale
     for the display axes.
  3. SPRING FORCE is drift, not forcing (see FORCING). Reconcile in the UI.
  4. NOISE CONVENTION -- pin g = sqrt(2D); additive-only in v1, state-dependent (multiplicative) later.
  5. SECTION COMPATIBILITY -- the model appears in every dropdown, but FDT/SBI need extra work and Reduction
     is excluded. Gate/disable a user model where unsupported with a clear message (mirror fdt_panel.py:34-45
     "FDT currently only supports NADROWSKI").
  6. NUMERICAL STABILITY -- fixed-step Euler-Maruyama, no adaptivity; arbitrary nonlinear systems can diverge.
     Add NaN/blow-up detection (reuse the N_ND_MAX prefilter idea) + a user-set dt/duration.
  7. SANDBOX the parser (locked sympy namespace; no eval of arbitrary Python). Low risk on a local desktop
     app, but still do it.
  8. FIRST-ORDER ONLY -- one dx/dt per variable; a 2nd-order system must be written as two first-order eqs.
     The UI enforces this (one row per variable).
  9. Optional: a Settings section documenting the 3 default models' equations (they're code, not
     parser-derived, so this is hand-written prose, not a round-trip).
