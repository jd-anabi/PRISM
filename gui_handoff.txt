GOAL
  A PySide6 desktop GUI as an ADDITIONAL front-end to the existing CLI research app.
  The CLI (`python -m core`) and the 10 diagnostic scripts in scripts/ MUST keep working
  unchanged — every core refactor done for the GUI is non-breaking and defaults to the old

DESIGN DECISIONS (already made + implemented; don't re-litigate)
  - Toolkit: PySide6. matplotlib backend is forced to Agg before any core.* import.
  - Style: a STAGED GUI that drives the already-decomposed orchestrator stage functions
    directly (Config -> Prior -> Posterior -> Validate -> Infer), one button per stage.
    NOT a modal-dialog port of the CLI prompts.
  - Heavy work runs on a background QThreadPool worker; the UI never blocks.
  - Figures are rendered to PNG on the worker and shown as images (see GOTCHA #2).
  - Progress (tqdm) renders in its OWN widget, one row per live bar. It never touches the log.

PROGRESS — how tqdm is handled (this was the top bug; it is fixed)
================================================================================
  THE BUG (was): tqdm bars appended hundreds of rows to the log pane instead of redrawing one line.

  THE ACTUAL ROOT CAUSE (established from tqdm 4.67.1's source + a byte-capture of its writes; the
  original handoff's five hypotheses were mostly wrong — (b) was outright refuted, and (a), the one
  it called "most likely", was NOT the cause):
    tqdm redraws a bar at pos>0 as THREE atomic writes (std.py:1493-1497, :451-460, :1441-1443):
        write("\n" * pos)                 # moveto(pos)
        write("\r" + frame + padding)     # print_status
        write("\x1b[A" * pos)             # moveto(-pos)   <- colorama IS installed, so real ANSI
    The third write has no terminator, so `frame + "\x1b[A"` stranded in the old reader's buffer, and
    the NEXT redraw's leading "\n" flushed it through the NEWLINE branch — i.e. as a LOG LINE. Each
    nested-bar redraw appended exactly one row and emitted ZERO progress signals. The pipeline nests
    bars FOUR deep, so this fired constantly.

  KEY INSIGHT OF THE FIX: every chunk tqdm writes is atomic and self-describing, so classify per
  chunk, not by scanning for terminators. There is no cursor state and thus nothing to desync.
  A paint's row index is NOT just "newlines in the preceding chunk" — moveto(+1) and close(leave=True)'s
  trailing "\n" (std.py:1303) are byte-identical. The authority is the chunk that FOLLOWS:
      a paint is at row n  <=>  it is followed by a pure up-move of n     (else row 0)
  A paint with no preceding newline is unambiguously row 0 and is published at once; a paint with
  preceding newlines is a guess and is held for exactly one chunk (its confirming up-move arrives
  microseconds later, in the same display() call).

  FILES
    core/gui/vt.py                    NEW. The whole classifier. Qt-FREE and unit-testable by replaying
                                      tqdm's literal chunks. RowState / parse_bar / StreamRouter.
    core/gui/streams.py               REWRITTEN. _SignalStream feeds a StreamRouter (stdout and stderr
                                      get SEPARATE routers). A _Pump daemon thread coalesces and emits
                                      to Qt at 15 Hz.
    core/gui/widgets/progress_pane.py NEW. One _BarRow per live bar + one overall QProgressBar.
    core/gui/widgets/log_pane.py      append_lines(batch); replace_last_line/_last_was_progress DELETED.
    core/gui/worker.py                progress=Signal(str) -> rows=Signal(object) + log_batch=Signal(object).

  *** THINGS THAT WILL BITE YOU IF YOU TOUCH core/gui/vt.py OR streams.py ***
   P1. DO NOT give _SignalStream a `fileno()` or an `encoding` attribute. Without them tqdm's
       screen-shape probe fails, so ncols is None (frames are NEVER disp_trim'd, which the frame regex
       relies on) and nrows is None (so every nested bar displays). Adding `encoding` also flips
       `ascii` (std.py:1038) and changes the glyphs.
   P2. A bare "\n" chunk is THREE different things: the terminator of a status line, the terminator of
       a plain print() (print writes its text and its "\n" as two separate chunks!), or a real
       moveto/finalizer. Only the last is cursor motion. Mistaking a print()'s terminator for a moveto
       strands the line forever and shifts the next bar down a phantom row. There is a test for this.
   P3. A bare "\r" is ALWAYS row 0 — tqdm only emits one when pos == 0 (std.py:1307).
   P4. The pump's FINAL publish must happen ON THE PUMP THREAD (stop() sets a flag and joins). If you
       emit inline from the caller's thread and that caller is the GUI thread, Qt resolves it as a
       DirectConnection and those slots run AHEAD of ticks still queued in the event loop — which
       silently scrambles the log into teardown-first order. This bit us once already.
   P5. The 15 Hz pump is LOAD-BEARING, not a nicety. tqdm's `mininterval` does not bound the redraw
       rate here: set_description() and reset() both call refresh() -> display() with NO time gate, and
       core/SBI/Priors/{bp,hopf,nadrowski}_prior.py call both on EVERY sweep iteration.
   P6. Rows are keyed by tqdm `pos`, NOT by desc. The prior sweeps rewrite desc with a live counter
       every iteration; a desc-keyed row map would mint a row per iteration — the original bug, reborn
       inside the new widget. A digit-normalised `ident` is used ONLY to detect "a different bar took
       this slot".
   P7. The overall bar tracks the DEEPEST live row whose total > 1. Not the outermost (the pos-0
       "Training neural posterior" bar wraps range(TRAINING_NUM_ROUNDS) and that is 1, so it reads 0%
       for the whole multi-hour build), and not a sticky first-seen driver (sbi's NN training emits NO
       tqdm bar at all — only a printed epoch counter — so a sticky driver sits at 100% through the
       longest phase). It falls back to indeterminate when nothing informative is live.
   P8. close(leave=True) at pos 0 paints the final frame and then writes a bare "\n" — byte-identical
       to a moveto(+1). If you just treat it as a moveto, the FINISHED bar stays in the pane at 100%
       and (being `informative`) pegs the overall bar at 100% while the pipeline is still working.
       StreamRouter._settle_closing() resolves the ambiguity on later evidence: assume moveto, but
       remember row 0 may be dead, and retire it once a chunk arrives proving no moveto was coming.
       BOTH real leave=True bars are pos 0 (prior.py:88, campaigns.py:214), so this is the common case.

SOLVER PERFORMANCE METER + LIVENESS  (core/gui/widgets/progress_pane.py)
================================================================================
  THE PROBLEM. A top-level iteration ("Generating training data") takes ~10 SECONDS, so the overall bar
  sits still between ticks and the GUI reads as frozen. The thing that IS moving is the SDE solver
  underneath it, and its it/s is the number the user wants to see.

  THE PANE now renders:
      Solver Performance: ++++  (13.3k it/s)   [▓▓▓▓▓▓▓░░░░░░]   <- rate meter + live step strip
      [███████░░░░░░░░░░░░░  38%]  ⠹                             <- overall bar + spinner
          Training neural posterior
          Generating training data  —  1902/5000 [05:12<13:41]

  *** FOUR THINGS THAT WILL BITE YOU HERE ***
   S1. The solver bar is found by its DESC PREFIX (config.SOLVER_BAR_DESC -> "step (batch="), NEVER by
       its row. Its tqdm `pos` is 0, 1 OR 2 depending on the phase and the panel: under a posterior build
       it sits beneath two other bars; during FDT Campaign 1 it is the only bar there is. The prefix is
       unique repo-wide (sdeint.py is the sole producer).
   S2. The solver bar must NEVER become a progress row. A posterior build constructs 10k–30k of them —
       one per time segment, each alive 1–10s — so a row would mean creating and destroying a widget
       every few seconds. ProgressPane.set_rows() partitions it out into the fixed solver line.
   S3. The solver bar must be EXCLUDED from _retarget(). Its total is in the tens of thousands and it is
       the deepest bar, so it would win the "deepest informative row" election every time and drag the
       overall bar through a full 0→100% sweep every second — instead of showing the top-level count.
   S4. tqdm flips to `s/it` BELOW 1 it/s (std.py:557), so " 2.50s/it" means 0.4 it/s, NOT 2.5.
       vt.parse_rate() inverts it. Read naively, a crawling solver would show MORE plus signs the slower
       it got. Also: a bar shorter than mininterval=1.0 emits only "?it/s" and, with leave=False, there
       is no final 100% frame — so rate samples arrive in GAPS and the last one is HELD (SOLVER_IDLE_S).

  THE METER. plus_meter(rate) = one '+' per order of magnitude: 10_000 it/s -> "++++". Under 10 it/s
  there is no order to show, so it renders '·' rather than an empty string. After SOLVER_IDLE_S (45s)
  with no sample the meter reads "— (idle)" — during NN training the solver genuinely is not running, and
  a stale "++++" would be a lie. The Reduction panel is purely analytical, so its meter is always idle.

  THE SPINNER is a braille QLabel on a 100ms QTimer. It is NOT purely decorative: ProgressPane.heartbeat()
  is stamped from every rows snapshot AND every log batch (BasePanel wires log_batch -> heartbeat), and
  after STALL_S (45s) of total silence the spinner FREEZES and reads "⏳ no output for 62s". A spinner
  that keeps twirling on a wedged run asserts progress that is not happening.

  DEFECTS FOUND BY AN ADVERSARIAL REVIEW AND FIXED (each has a regression test — don't undo them)
   - A cell dropped into Resources/Cells/<model>/ with NO sibling Bounds/<model>/<same>.txt made
     cli._parse_cell raise a BARE ValueError (not UnitParseError). CrossValPanel prefills from
     _parse_cell in __init__, so it escaped MainWindow() -> build_app() and the WHOLE GUI failed to
     launch — before app.py's excepthook was even installed. Panels now catch broadly
     (BasePanel._config_error); cli's builders raise plain ValueErrors, so `except cli.UnitParseError`
     is simply the wrong net.
   - Every dispatch() pinned its Worker's ARGUMENTS (cfg, prior, posterior, CUDA tensors) forever:
     QThreadPool.start() hands the QRunnable to C++ and setAutoDelete(False) means C++ never frees it,
     and the _finished closure captures the worker in a cycle. _finished now clears fn/args/kwargs.
   - streams._REDIRECT was acquired OUTSIDE the try/finally, so a _Pump() thread-start failure would
     hold it for the whole process and every later run would silently lose ALL its output.
   - Closing the window mid-run tore down the QApplication under the worker -> "RuntimeError: Signal
     source has been deleted", plus a silent process-wide block until the run finished. The emits are
     now guarded and MainWindow.closeEvent asks first (a run still cannot be cancelled — Phase 3).
   - set_controls_enabled now locks the WHOLE left column, not just the run button: ArtifactPicker
     .refresh() wraps list_dir in contextlib.redirect_stdout, which reassigns the PROCESS-WIDE
     sys.stdout — i.e. the stream redirect_streams installed for the running worker. Clicking ⟳ or a
     model combo mid-run would swallow the worker's output, and could permanently reinstate a dead
     _SignalStream as the process's stdout.

  CORE CHANGE (the only behavioural core change in this whole effort)
    core/config.py: QUIET_SEGMENT_BAR = False.  core.gui.app.build_app() sets it True.
    ONE call site passes `**({"disable": True} if config.QUIET_SEGMENT_BAR else {})`:
      core/Simulator/simulator.py    (the per-time-segment bar — it wraps segs in {1,2,3}, useless)
    Under the GUI this takes the Posterior nest from 4 deep to 3 (a disabled bar surrenders its
    nesting slot — std.py:985-992 removes it from _instances).
    !! Do NOT "simplify" this to `disable=config.QUIET_SEGMENT_BAR`. tqdm.__init__ is
       @envwrap("TQDM_")-decorated (std.py:951) and a call kwarg OUTRANKS the environment, so an
       explicit disable=False would shadow a TQDM_DISABLE override that reaches that site today.
       The conditional splat means the CLI passes no `disable` kwarg at all — byte-identical.
    Import it BY MODULE (`from core import config`), never `from core.config import QUIET_SEGMENT_BAR`

*** GOTCHAS — READ THESE BEFORE TOUCHING THE THREADING/FIGURE CODE ***
  1. WORKER LIFETIME (cost hours to find). A Worker created as a local in dispatch() gets garbage
     collected as soon as run() returns, and Qt then PURGES that sender's still-queued result/finished
     events — the slots never fire and the panel stays busy=True forever.
     FIX (in BasePanel.dispatch): worker.setAutoDelete(False) AND keep it in self._workers until the
     finished slot discards it. NEVER start a raw Worker outside BasePanel.dispatch.
  2. NEVER PAINT A WORKER-CREATED MATPLOTLIB FIGURE. Wrapping a figure a worker thread made with pyplot
     in a live FigureCanvasQTAgg and letting a *shown* widget paint it DEADLOCKS on matplotlib's global
     lock (silent hang, no traceback).
     FIX: BasePanel._png_fig_sink renders the Figure to PNG bytes ON THE WORKER THREAD and emits the
     bytes; FigureStack shows a QPixmap. Figures are therefore STATIC images (no zoom/pan). Accepted.
  3. Agg BACKEND is forced in core/gui/__main__.py BEFORE any core.* import. That is what makes it safe
     to run the runners on a worker thread: every un-refactored plt.show() becomes a harmless no-op.
     Do not move/remove that matplotlib.use("Agg") line.
  4. redirect_streams swaps sys.stdout/stderr PROCESS-WIDE. Therefore ONLY ONE PANEL MAY RUN AT A TIME,
     app-wide — BasePanel._running is a CLASS attribute for exactly this reason (a per-panel guard would
     let the FDT tab start while SBI is training and the two would fight over the console).
     streams._REDIRECT is the backstop: a second redirect declines and warns rather than corrupting
     sys.stdout permanently.
  5. Resources/ layout is per-model subfolders:

PHASE 3 — cancellation, QSettings, better error dialogs  (DONE)
================================================================================
  CANCELLATION is COOPERATIVE and needs ZERO core changes. The handoff used to say a mid-train kill was
  impossible without a subprocess — that was WRONG. Every print() and tqdm redraw funnels through
  _SignalStream.write() on the worker thread, and sbi prints an epoch counter INSIDE its fit loop
  (npe_base.py:437, show_progress_bars defaults True), so a flag checked in write() is a checkpoint
  reaching even inside sbi's training loop.
    - streams.CancelToken (requested: Event, fired: bool) + WorkerCancelled(BaseException). BaseException
      so it sails through the pipeline's many `except Exception`; caught by name only in Worker.run.
    - _SignalStream.write() calls cancel.check() BEFORE feeding the router; check() LATCHES (raises once)
      so tqdm's teardown writes can finish and the final bar frames still reach the log.
    - BasePanel: a per-dispatch token, a Cancel button (hidden until a run), the class attr _active_cancel
      + request_cancel_all() for MainWindow.closeEvent's "Cancel task & quit".
    - Worker.run catches WorkerCancelled -> emits `cancelled` (NOT `error`; a cancel is not a failure) and
      plt.close("all") on the way out.
    Latency: ~1s almost everywhere (the solver bar's mininterval=1.0), but up to ONE NN-training epoch
    (~10-60s) or the check_sbc C2ST block (~46s) — those are silent, so the button says "Cancelling…".

  *** TWO TQDM GOTCHAS (each cost real time; each has a regression test) ***
   C1. tqdm's refresh() (tqdm/std.py:1346-1349) does a MANUAL self._lock.acquire()/display()/release(),
       NOT a `with`. Our cancel raises from inside display()'s write(), so the release is SKIPPED and
       tqdm's global write lock LEAKS. The next tqdm.__new__ then DEADLOCKS acquiring it — i.e. cancel a
       run, start another, hang. redirect_streams' finally calls streams.reset_tqdm_lock() when
       cancel.fired (on the worker thread, always) to abandon the leaked lock and install a fresh one.
       Do NOT remove that. test_cancel_token_latches_and_leaves_tqdm_usable pins it (it HANGS without).
   C2. tqdm runs a TMonitor DAEMON thread that force-refreshes a bar gone quiet for >=maxinterval (10s),
       writing to our stream FROM ITS THREAD. If that write consumed the cancel latch it would raise
       where nobody catches it (killing the monitor) and leave the worker to sail past a fired latch —
       silently LOSING the cancel for the whole run. So CancelToken.check() only raises on the OWNER
       thread: redirect_streams calls cancel.arm() (on the worker thread) to record the owner, and a
       write from any other thread does not consume the latch. test_cancel_is_not_consumed_by_a_non_
       worker_thread pins it.

  QSETTINGS — core/gui/settings.py (one store; app.py sets BOTH org + app name, required on Windows).
    Each panel has save_settings(qs)/restore_settings(qs); MainWindow saves all + window geometry on an
    ACCEPTED close only (never on a cancelled close), and each panel restores at the END of its __init__.
   *** RESTORE-ORDER TRAPS ***
    - SBI/FDT: restore model FIRST and call _on_model_changed(model) explicitly (currentTextChanged does
      NOT fire if the value already equals the default), THEN the pickers — a picker restored first gets
      wiped by the model's refresh().
    - CrossVal: restore preset + cell first; DO NOT persist the cell-derived s/t grid lo/hi (they are
      re-derived from the cell file; a saved value from a DIFFERENT cell is a stale, wrong bound). Only
      the free knobs persist (preset, cell, f0, freqs_per_batch, grid points).
    - Pickers: persist combo.currentData() (the model-namespaced relpath); restore via findData with a
      -1 guard (file gone -> leave at default, never setCurrentIndex(-1)). ArtifactPicker.key()/restore_key().

  ERROR DIALOGS: app.py installs the excepthook BEFORE MainWindow() (a panel-construction failure now
    shows a dialog, not a bare console traceback) and wraps construction in a startup-failure dialog.

    _config_error uses a Warning icon + "Check your inputs" (a bad file pick is input, not a crash).

SIMULATE SECTION — real-time streaming (the 4th home button; BUILT)
================================================================================
  Pick a model + cell + observation time, press Start, and watch a live scrolling trace of the
  hair-bundle displacement (the FIRST state variable in every model: Nadrowski x0, Hopf x0, BP/BPSteady
  x_hb) over a top-down "hair bundle" heatmap. A worker-thread chunked Euler loop streams numpy frames to
  a GUI-thread pyqtgraph view. Files: panels/simulate_panel.py + panels/simulate_runner.py +
  widgets/live_hair_bundle.py. Reuses the BasePanel dispatch rails verbatim; the ONLY new transport is a
  `chunk` signal + dispatch(provide_stream=). requirements.txt pins pyqtgraph==0.13.7.

  CONFIG (simulate_runner.build_stream_config). No separate bounds pick: the bounds file is the per-model
  SIBLING of the cell (Bounds/<model>/<cell>.txt), so build make_sim_config(sibling) + load_and_validate_gt
  (mirrors inference_tabs._run_simulated_preview). Then cfg.hw = cpu_device().
    !! cfg.hw = cpu_device() is REQUIRED. make_sim_config defaults hw=detect_device() (CUDA on a capable
       box), but the batch-1 loop is CPU-optimal AND every tensor must share one device; sim.sde.force =
       force_chunk is a bare attribute set with no .to(device), so a CUDA/CPU mix would raise mid-stream.

  THE STREAMING LOOP (simulate_runner.run_simulation_stream). Does NOT reuse generate_observations (it
  allocates the ~2.4M-point cfg.t and simulates the whole horizon at once). It mirrors the segment loop in
  simulator.py:34-68 incrementally, driving Solver().euler DIRECTLY, one frame_steps chunk per frame:
    - plan_stream(cfg, T_obs) is PURE: forcing_gt/rescale_gt, t_scale_gt, subsample_factor, steady_steps,
      total_steps = steady_steps + N_obs*subsample_factor -- the SAME arithmetic as generate_observations
      (orchestrator.py:120-133), so a streamed trace matches the pipeline's observation. Bounds a FINITE
      stream.
    *** THINGS THAT WILL BITE YOU ***
     SIM1. Construct the VALID_SIMS[model] Simulator ONCE, up front, GUARDED. Its _set_up_model calls
           exit() on a bad construction -> SystemExit, a BaseException that Worker.run does NOT catch
           (silent worker-thread death). _make_simulator catches SystemExit -> RuntimeError so the panel
           shows a dialog. Driving Solver().euler directly (not Simulator.simulate/__sols) also dodges
           __sols' exit().
     SIM2. dt CONTINUITY: a frame of m steps needs m+1 grid points (frame_time_grid) -- euler derives
           dt=(t1-t0)/(n-1) from linspace(t0,t1,n). m points would make dt != dt_nd_min (silent timescale
           bug). res[0] duplicates the boundary, so emit res[1:]; advance t_now by m*dt_nd exactly.
     SIM3. torch.no_grad() around the loop is REQUIRED -- Solver().euler has no autograd guard of its own
           (that lived in Simulator.__sols, which we bypass); an unbounded stream would leak the graph.
     SIM4. CANCEL: dispatch injects should_stop = self._cancel.requested.is_set. The runner polls it once
           per frame and raises WorkerCancelled BETWEEN frames -- never inside a tqdm redraw, so no tqdm
           write-lock leak (cancel.fired stays False; reset_tqdm_lock is skipped, correctly). Reuses the
           `cancelled` UI path. Latency ~one frame. Wall-clock-paced with time.sleep on the WORKER thread.
     SIM5. sim.sde.force = force_chunk (bare attribute set, mirroring simulator.py:58) -- NOT sim.force=,
           which rebuilds the whole model. Hopf auto-gets 2 force channels (build_nondim_sin_force_tensor
           emits 2 iff "amp_y" in forcing_idx). Use sim.inits as the first curr_inits (BP-steady slices it
           to 4). Nadrowski passes state_dep_drift=True.

  TRANSPORT. worker.py adds chunk=Signal(object). base_panel.dispatch(provide_stream=True, on_chunk=)
  injects emit_chunk=worker.signals.chunk.emit + should_stop, and connects chunk->on_chunk (copies the
  provide_fig_sink pattern). Each emitted frame is a FRESH (k,2) [t_seconds, x_displacement] array
  (Signal(object) marshals by reference; never reuse a buffer). t is in SECONDS (one display sample =
  DT_EXP_S apart, unit-independent); x is redimensionalized via helpers.rescale(hb_nd, x_scale, x_offset).

  RENDER (widgets/live_hair_bundle.py, GUI THREAD). GOTCHA #2 (never paint a worker-built matplotlib
  figure) is matplotlib-specific; pyqtgraph items are GUI-owned, so on_chunk=LiveHairBundleView.push
  updates them safely. The panel mounts the view via self.right_layout.insertWidget(0, view, 5) and hides
  the (static) figure stack. The trace is a fixed-size ring buffer (PlotDataItem.setData); the HEATMAP is
  an anisotropic Gaussian -- an ELLIPSE whose MAJOR axis is perpendicular to the oscillation (displacement
  slides the blob horizontally, so sigma_perp(vertical) > sigma_par(horizontal)), over a WIDE field
  (aspect~2.6, aspect-locked) with the center mapped into a horizontal MARGIN so it never clips at the
  oscillation peaks. gaussian_field(cx, cy, gx, gy, sx, sy) is a pure, unit-tested function; ImageItem uses
  fixed levels=(0,1) so there's no per-frame autoscale flicker.

  VIDEO EXPORT ("Save video…", panels/simulate_export.py). The panel RECORDS each emitted (k,2) frame
  during a run (SimulatePanel._on_chunk wraps push + append; reset in _start). "Save video…" (enabled once

  export_animation on the worker. It is a RE-RENDER, not a screen-grab: the live stream emits only a few
  coarse chunks, so the exporter sweeps the fine display series with export_stride(1/DT_EXP_S, video_fps)

   *** THINGS THAT WILL BITE YOU ***
    V1. buffer_rgba()[...,:3] is a NON-contiguous view (tostring_rgb is gone in mpl 3.10); wrap in

USER-DEFINED MODEL BUILDER + SETTINGS ADDITIONS  (shipped 2026-07-17)
================================================================================
  WHAT IT IS. A builder where a user declares state variables and, per variable, types the deterministic
  RHS of dx/dt (nondimensional, first-order, may be nonlinear), a white-noise strength D (the solver
  diffusion is g = sqrt(2*D); parameters only -> additive noise), and OPTIONALLY one time-driven forcing
  from {sinusoidal, step, triangular, exponential}. The model is parsed by sympy, compiled to torch,
  SAVED to Resources/Models/<NAME>.json (+ a generated Bounds/Cells/Units triple), registered at startup,
  and appears in every model dropdown. v1 is SIMULATE-ONLY: user models are gated OUT of FDT / Parameter
  Inference / Reduction (the SBI stack has no Prior/INIT_SHAPES for them, and FDT hardcodes Nadrowski's
  T_eff params). The forward roadmap to SBI-ready (v2) and FDT (v3) user models is in features_handoff.txt.
  Reached from Settings -> "User-defined models" -> "Open model builder" (nav screen index 6; not a Home
  tile). Observable = state column 0 (same as every built-in model).

  NEW MODULES (all in core/, NOT core/gui/ -- the science/runtime layer):
    registry.py                 ModelSpec + a name-keyed registry. The 3 built-ins are seeded at import
                                (light -- no Simulator imports). register(spec) APPENDS user models to the
                                live config.VALID_MODELS / VALID_LABELS lists (built-ins keep indices 0-2;
                                an assert enforces it). load_user_models() is an EXPLICIT call (build_app +
                                core/__main__), NEVER an import side effect -- the test suite builds
                                MainWindow directly and must see pristine registries. Per-file load failures
                                land in registry.load_errors (shown on the Settings screen), never raised.
    Models/user_model.py        parse_user_model(variables) -> CompiledUserModel; UserModel (the ONE generic
                                torch model class every user model shares -- mirrors the hopf_model duck-typed
                                contract: f(x,t) with the integer step index + additive self.force[:, j, t];
                                g() returns the cached (batch,d) sqrt(2D) diagonal vector). SANDBOXED: a
                                token pre-filter + a locked parse namespace + a hand-rolled tree->torch
                                compiler (_compile_tree) -- no eval of arbitrary Python.
    Simulator/user_simulator.py UserSimulator(Simulator): generic _set_up_model that builds UserModel via
                                Model(*torch.unbind(params, dim=1), force, ...); RAISES RuntimeError on
                                failure (no legacy print+exit()). Sets self._compiled BEFORE super().__init__.
    forcing.py                  build_nondim_force_tensor(..., kind=) implementing all 4 carriers, +
                                build_user_force_tensor(spec, ...) -> (batch, n_vars, T). LIGHT (torch/numpy/
                                helpers only, no sbi import) so the GUI + tests import it freely.
                                pipeline.build_nondim_sin_force_tensor now DELEGATES here (kind="sin") --
                                numerically identical, pinned by a golden test (SBI training data depends on it).
    Helpers/model_store.py      save_user_model(doc)/load_user_model/list_user_models/delete_user_model +
                                validate_name. Writes the JSON AND emits Bounds/<name>/default.txt +
                                Cells/<name>/default.txt + Units/<name>/units.txt so cli.make_sim_config /
                                _parse_cell / simulate_runner.build_stream_config consume a user model exactly
                                like a built-in (they resolve by folder name).

  THE FORCE-CHANNEL CONVENTION for user models: ONE force row per state variable, (batch, n_vars, T),
  zeros where unforced; UserModel.f adds force[:, j, t] to variable j. Forcing param names are suffixed
  <pname>_<var> (amp_x, t0_x, ...) so they flow through the existing name-keyed forcing_idx machinery.
  (Built-ins keep their legacy 1-or-2-channel sinusoidal convention -- 2 iff "amp_y" in forcing_idx.)

  SIMULATE INTEGRATION (simulate_runner.py). build_stream_config resolves labels + state_dep_drift via
  the registry (state_dep_drift is ALWAYS False for user models -- D is params-only in v1); plan_stream
  sets user_spec + n_channels = spec.n_vars; _make_simulator branches to registry.make_user_simulator;
  run_simulation_stream builds the per-frame force via forcing.build_user_force_tensor. A per-frame
  BLOWUP_ND_LIMIT (1e6) + isfinite check after solver.euler raises a readable "diverged near t..." error
  (arbitrary user drift can blow up under fixed-step Euler-Maruyama).

  PERSISTENCE / JSON schema (schema_version 1): {name, variables:[{name, drift, D, init, forcing:
  null|{kind, params, sign}}], params:{name:value}, rescale:{x_scale, t_scale}}. The generated Bounds ND
  section order == parse_user_model's param discovery order == the UserModel constructor column order (the
  LOAD-BEARING positional contract). Placeholder ND bounds are (v-max(|v|,1), v+max(|v|,1)); x_scale/t_scale
  bounds are (v/2, 2v) -- STRICTLY POSITIVE because SimConfig.t_nd_max divides by the t_scale LOWER bound.
  Units are fixed "nm s" (x_scale in nm, t_scale in s).

  MainWindow wires these: _open_model_builder/_edit_user_model/_delete_user_model (delete confirms + is

   *** THINGS THAT WILL BITE YOU (each is a fixed adversarial-review finding + has a regression test) ***
    U1. PARAM DISCOVERY must ignore numeric literals. _identifiers strips numbers (incl. "1e-3"/"0x1F")
        BEFORE the identifier scan, else the mantissa tail ("e", "e3", "x1F") becomes a phantom parameter
        -- a dead column that desyncs the positional contract. parse_user_model also keeps ONLY parameters
        the parsed expressions actually reference (sympy can simplify some away).
    U2. "E" is an ORDINARY parameter, not Euler's number (physics names E are common); only "pi" is a
        constant. Names that shadow parse_expr's own constructors (Float/Integer/Rational/Symbol/Function)
        are REJECTED with a clear message (they would break every numeric literal otherwise).
    U3. build_stream_config re-checks that the model's compiled param_names EQUAL the emitted Bounds file's
        ND key order, and raises "out of sync" if a hand-edited JSON drifted from a stale Bounds file --
        else torch.unbind would silently mis-bind values by position (wrong physics, no error).
    U4. save_user_model REFUSES: non-finite values (inf/nan -> unreadable tokens), t_scale >= the transient
        budget ceiling (T_SCALE_MAX_S; a slow model would die on SimConfig.steady_idx's assert AFTER a clean
        save), Windows reserved device names (NUL/CON/COM1.., whose folders half-create), and built-in/
        existing names. It writes the triple FIRST and the JSON LAST (the registry loads from the JSON, so
        that is the commit point); delete_user_model removes the JSON LAST for the same reason.
    U5. _refresh_model_combos re-fires a panel's _on_model_changed ONLY when the selection actually changed.
        The hooks refresh the cell/bounds pickers (which resets them to their first entry), so firing on a
        mere item-list update would silently discard the user's picker selections app-wide on every save/
        delete. A deleted selected model still falls back (to NADROWSKI) and re-fires correctly.
    U6. The builder's Validate/Save run a short (100-step, batch-1) smoke integration ON THE GUI THREAD --
        cheap, so no BasePanel.dispatch. But its solver's tqdm writes to the PROCESS-WIDE redirected streams,
        so _validate/_save REFUSE while BasePanel._running (a worker owns those streams). The screen is a
        plain QWidget (not a BasePanel), so it is NOT auto-locked by the app-wide control lock -- this guard
        is the substitute.

  NOT yet exercised on a real display (ask before assuming): the full builder round-trip on screen (declare
  vars -> per-variable rows -> Detect parameters -> Validate -> Save -> the model appears in Simulate ->
  stream a sane trace), the Edit/Delete flows, the FDT/Inference gate MESSAGES, and the accent/Inter
